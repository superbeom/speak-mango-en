# Documentation Maintenance Workflow

이 워크플로우는 프로젝트의 변경 사항(`git status`)을 분석하여 관련 문서들이 최신 상태를 유지하도록 업데이트합니다.

## 1. Check Changes (변경 감지)

1.  `git status` 명령어를 실행하여 현재 변경되거나 Staged된 파일 목록을 확인하십시오.
2.  코드 변경 사항이 있다면 다음 문서들 중 업데이트가 필요한 항목을 식별하십시오.

## 2. Verify & Update Documents (문서 검증 및 업데이트)

**⚠️ 중요: 아래 명시된 모든 문서를 하나씩 점검하여 누락 없이 현행화하십시오.**

다음 기준에 따라 각 문서를 점검하고, 내용이 누락되었다면 업데이트하십시오.

- **`docs/project_context.md`**:
  - 프로젝트 구조, 아키텍처, 규칙, 기술 스택 변경이 있었나요?
- **`docs/project_history.md`**:
  - 중요한 기능 추가, 버그 수정, 아키텍처 변경, 의사결정이 있었나요?
  - 있다면 날짜와 함께 항목을 추가하십시오.
- **`docs/walkthrough.md`**:
  - 사용자에게 보여지는 기능이나 UI/UX 변경 사항이 있나요?
  - 구현 상세 내용(버전)을 기록하십시오.
- **`docs/task.md`**:
  - 완료된 작업(`Task`)이 있다면 체크박스를 `[x]`로 변경하십시오.
  - 새로운 작업이 발견되면 목록에 추가하십시오.
- **`docs/future_todos.md`**:
  - 작업 중 발견한 기술 부채나 추후 개선 사항이 있나요?
- **`docs/features_list.md`**:
  - 기능의 동작 방식이나 스펙이 변경되었나요?
  - 최신 스펙으로 업데이트하십시오.
- **`docs/agent_workflows.md`**:
  - 에이전트 워크플로우 파일(`*.md`)이 추가되거나 수정되었나요?
- **`docs/supabase_strategy.md`**:
  - 데이터베이스 운영 전략이나 스키마 관리 방식이 변경되었나요?
- **`docs/database_schema.md`**:
  - 데이터베이스 스키마나 테이블 명세에 변경 사항이 있었나요?

## 3. Stage Changes (변경 사항 반영)

1.  문서 업데이트가 완료되면, 변경된 문서 파일들을 `git add <file>` 명령어로 Staging Area에 추가하십시오.
2.  문서 변경 사항이 없다면, 사용자에게 알리고 이 단계를 건너뛰십시오. (단, 문서 업데이트가 필요한 상황임에도 변경되지 않았다면 다시 검토하십시오.)
